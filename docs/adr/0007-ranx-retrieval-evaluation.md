# 0007. Golden-set retrieval evaluation with ranx

- **Status:** Proposed
- **Date:** 2026-07-02
- **Deciders:** Maintainers
- **Related:** [Measuring Retrieval Relevance](https://qdrant.tech/documentation/improve-search/retrieval-relevance/), [Measuring ANN Recall](https://qdrant.tech/documentation/tutorials-search-engineering/ann-recall/), [ADR 0003](0003-hybrid-search-rrf-default.md), [ADR 0006](0006-explicit-fastembed-pipeline.md)

## Context

Qdrant’s [Improve Search](https://qdrant.tech/documentation/improve-search/) track treats retrieval quality as a **four-layer evaluation stack**:

1. **ANN recall** — does approximate HNSW search match exact kNN?
2. **Retrieval relevance** — do top-k hits match query intent? ([Measuring Retrieval Relevance](https://qdrant.tech/documentation/improve-search/retrieval-relevance/))
3. **Pipeline output quality** — does retrieval + consumer produce correct answers? ([Evaluating Pipeline Output Quality](https://qdrant.tech/documentation/improve-search/pipeline-output-quality/))
4. **Business impact** — application KPIs (out of scope for the MCP server)

Layer 2 is directly applicable. The [Measuring Retrieval Relevance](https://qdrant.tech/documentation/improve-search/retrieval-relevance/) tutorial uses **ranx** to compare labeled **Qrels** (ground truth) against a **Run** (actual retrieval output) with metrics such as `recall@k`, `MRR`, and `NDCG@k`.

Today we measure **latency and throughput** in `benchmarks/bench.py` (p50/p95 search latency, indexing chunks/sec, payload-index A/B) but not **ranking quality**. We cannot objectively answer:

- Does hybrid RRF beat dense-only on symbol-name queries?
- Did a chunking or embedding change improve recall@10 on real code questions?
- Is `PREFETCH_MULTIPLIER=5` sufficient for our golden set?

Without a golden set keyed to **`chunk_id`** (existing stable ID per [ADR 0006](0006-explicit-fastembed-pipeline.md)), tuning hybrid search, HNSW params, or chunk strategy remains guesswork.

### Constraints

- Default deployment stays self-hosted; no cloud embedding APIs required for evaluation
- Evaluation runs **offline** (benchmark harness or CI job), not on every MCP tool call
- Golden set must use the same `chunk_id` keys stored in Qdrant payloads
- Complements — does not replace — latency benchmarks in `bench.py`

## Decision

We will add an **optional retrieval evaluation harness** that follows Qdrant’s ranx tutorial pattern, scoped to **layer 2 (retrieval relevance)** only.

### Golden set format

JSONL or YAML file checked into `mcp_server/benchmarks/fixtures/` (or generated by a maintainer script):

```json
{
  "query_id": "q_embedder_batch",
  "query_text": "adaptive batch sizing for dense embeddings",
  "collection": "codebase-indexer-mcp",
  "labels": {
    "mcp_server/src/codebase_indexer/indexer/embedder.py:42": 2,
    "mcp_server/tests/test_ollama_dense_backend.py:10": 1
  }
}
```

- **Keys in `labels`** are `chunk_id` values (`sha256("{rel_path}:{start_line}")`) or a documented alias resolved via payload scroll
- **Values** are graded relevance (0–2) for `NDCG@k`, or binary `1` for `recall@k` / `MRR`
- Queries cover code-search modes: symbol lookup, conceptual questions, cross-file behavior, config/env discovery

### Evaluation harness

New module `mcp_server/benchmarks/eval_retrieval.py`:

1. Load golden set
2. For each entry, call the same search path as MCP tools (`Embedder` + `QdrantStorage.search` with current `HYBRID_SEARCH` / prefetch settings)
3. Build ranx `Run` dict: `{query_id: {chunk_id: score}}`
4. Build ranx `Qrels` from labels
5. `evaluate(qrels, run, ["recall@10", "mrr", "ndcg@10"])`
6. Emit JSON report for CI comparison (same pattern as `bench.py --compare`)

**Optional dependency:** `ranx` as a dev/benchmark extra in `pyproject.toml` (`[project.optional-dependencies] benchmark = ["ranx"]`), not a runtime MCP dependency.

### ANN recall (layer 1)

Adopt Qdrant’s recommendation to check **ANN recall** via the Qdrant Web UI “Check Index Quality” tab or REST API when tuning `HNSW` / `hnsw_ef` after large re-indexes. No in-repo ANN recall implementation in phase 1 — document the operational step in `DEPLOYMENT.md`. Revisit programmatic ANN recall if bench regressions suggest index misconfiguration.

### Metrics defaults for code RAG

Per the tutorial’s metric guide:

| Scenario | Metric | Why |
|----------|--------|-----|
| MCP client reads top-k chunks | `recall@10` | Did the right chunk appear anywhere in the tool result set? |
| First result is the definition | `MRR` | Symbol / “where is X defined?” queries |
| Graded labels (primary + related) | `NDCG@10` | Rank quality when multiple chunks are partially relevant |

### CI integration

- **Not gated on every PR by default** — golden set maintenance is human-curated
- Optional workflow: run `eval_retrieval.py` on release branches; fail if `recall@10` drops more than a configured delta vs baseline JSON (mirrors `bench.py --compare`)

## Alternatives considered

| Option | Pros | Cons |
|--------|------|------|
| **ranx golden-set harness (chosen)** | Matches Qdrant Improve Search tutorial; standard IR metrics; A/B hybrid vs dense | Requires curated labels; not automatic |
| **Latency-only bench (status quo)** | Already implemented | No quality signal |
| **Manual spot checks** | Zero tooling | Not reproducible; regressions slip through |
| **Ragas context_precision only** | End-user framing | Needs LLM judge; blurs retrieval vs generation ([ADR 0010](0010-defer-ragas-to-client.md)) |
| **Live query logging + click feedback** | Production realism | No labels; privacy concerns for private repos |

## Consequences

### Positive

- Objective before/after comparisons when changing hybrid fusion, chunk size, embedding model, or prefetch multiplier
- Aligns with Qdrant’s documented tuning workflow and the production documentation search course evaluation step
- Uses existing `chunk_id` — no new ID scheme
- Pairs with [ADR 0010](0010-defer-ragas-to-client.md) 2×2 diagnostic (`recall@10` vs client-side faithfulness)

### Negative / trade-offs

- Golden sets drift as code moves; labels need periodic refresh
- Small golden sets may not represent all languages/repos
- `ranx` adds optional dev dependency
- RRF scores in the Run must use rank or fused score consistently — not comparable to cosine thresholds

### Neutral / follow-ups

- Script to suggest label candidates from `search_symbols` + maintainer confirmation
- Multi-collection golden entries (cross-project queries)
- `compare()` across multiple pipeline configs in one report (Qdrant course pattern)

## Implementation notes

### Affected paths

- `mcp_server/benchmarks/eval_retrieval.py` — new harness
- `mcp_server/benchmarks/fixtures/golden_queries.jsonl` — starter golden set on this repo
- `mcp_server/pyproject.toml` — optional `benchmark` extra with `ranx`
- `docs/DEPLOYMENT.md` — ANN recall check after HNSW tuning

### Rollout

Opt-in; default MCP behavior unchanged.

### Re-index

**No** for the harness itself. Label validity may require re-labeling after major chunking changes.

## Validation

- Unit test: mock search results → ranx metrics match hand-calculated recall@1 on a 3-query fixture
- Integration test (Qdrant required): golden set on indexed self-repo returns `recall@10 > 0` for smoke
- Manual: run harness before/after toggling `HYBRID_SEARCH` on fixture corpus; hybrid should improve symbol-query recall

Success criteria:

- Maintainers can run one command and get `recall@10`, `MRR`, `NDCG@10` JSON
- Hybrid vs dense-only delta is measurable on the checked-in golden set
- No new runtime dependencies in the MCP server image
